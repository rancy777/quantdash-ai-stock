# UI Design

QuantDash 的 UI 规则以“高密度信息 + 低噪音”优先，不追求炫技式仪表盘。

## 总体原则

- 先做信息层级，再做装饰。
- 优先复用现有 `GlassCard`、`Badge`、已有 toolbar / sidebar / panel 结构。
- 页面应尽量收口为 `page shell + workflow hook + panel/component`，避免把状态、数据读取和 JSX 堆在一个文件里。
- 不做无理由的主题改版。重构默认保持现有配色、圆角、阴影和排版语言。

## 布局规则

- 桌面端优先双栏或三段式布局，主内容区比控制区更宽。
- 窄屏下优先允许换行、纵向堆叠和横向滚动，不要让按钮或 tabs 被硬挤压。
- 工具条应紧凑，复杂筛选尽量折叠，不要长期占据主阅读区。

## 组件规则

- 标题卡片统一使用简洁标题和少量状态徽标，不在标题区堆过多说明文本。
- hover card、详情卡和统计卡沿用当前圆角和半透明玻璃风格，不单独引入新的视觉体系。
- 图表面板应与摘要面板拆分，保持单面板单职责。
- 列表区负责导航，详情区负责阅读，不要把同一批操作按钮重复放在两侧。

## 交互规则

- 所有 AI / 数据加载类操作必须有明确状态：`loading`、成功反馈、失败反馈。
- 复制、导出、保存等短反馈优先使用轻量 badge / inline feedback，不弹过重的模态。
- 历史记录切换、日期切换、指标切换都应保持“当前选中项”可见。

## 可维护性规则

- 新增复杂页面时，先拆 `hooks`、`panels`、`config`，不要再写超大组件。
- 公共样式常量、选项列表、标签映射优先放到 `config.ts(x)`。
- 页面内的临时格式化逻辑或反馈逻辑，超过一屏时应抽到 hook 或 helper。

## 禁止项

- 不引入与现有设计冲突的新配色体系。
- 不为“重构”顺手重写页面观感。
- 不把业务判断直接塞进展示组件。
